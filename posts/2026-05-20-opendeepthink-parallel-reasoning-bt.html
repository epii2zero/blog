OpenDeepThink: AI끼리 토론시키면 코딩 실력이 Elo 405점 뛴다

AI가 어려운 문제를 풀 때, 혼자 오래 생각하게 하는 것보다 여러 답안을 동시에 만들고 서로 비교·토론시키는 것이 훨씬 효과적이라는 연구 결과가 나왔다. UC 샌디에이고 연구진이 개발한 OpenDeepThink는 LLM(대규모 언어모델)의 답안들을 진화 알고리즘처럼 경쟁시켜, Gemini 3.1 Pro의 경쟁 프로그래밍(Codeforces) Elo 점수를 405점 끌어올렸다. 추가 학습 없이 추론 시점의 연산만으로 달성한 결과다.

연구 배경: '더 오래 생각하기'의 한계

최근 AI 연구에서 주목받는 전략 중 하나가 테스트 타임 컴퓨트 스케일링 — 모델이 답을 내기 전에 더 많은 연산을 투입하여 사고의 질을 높이는 접근법이다. '긴 사고 사슬(chain-of-thought)'이나 반복 자기 검증 방식이 대표적이다.

그러나 근본적 병목이 있다. 하나의 추론 경로를 아무리 깊게 파고들어도, 처음 잡은 방향이 틀렸으면 시간 낭비에 그친다. 여러 답안을 생성하고 최선을 고르는 'best-of-N' 방식도 있지만, 선택 메커니즘이 문제다. 기존에는 각 답안을 개별적으로 점수 매기는 '점수제 평가(pointwise scoring)'를 사용했는데, LLM 심판은 개별 평가에서 체계적으로 후한 점수를 주는 편향이 있어 좋은 답과 나쁜 답의 변별력이 떨어진다.

핵심 방법론: 진화적 토너먼트와 Bradley-Terry 랭킹

OpenDeepThink는 세 단계로 작동하는 '집단 기반(population-based)' 프레임워크다.

1단계: 초기 답안 생성

하나의 문제에 대해 20개의 후보 답안을 동시에(병렬로) 생성한다.

2단계: 진화 루프 (3세대)

각 세대에서 무작위로 답안 쌍을 짝지어 1

비교를 실시한다. 후보당 4번의 비교를 거친 뒤, 결과를 Bradley-Terry 모델(체스 Elo 레이팅과 같은 원리의 통계 모델)로 종합하여 전체 순위를 산출한다. 상위 25%(엘리트)는 그대로 보존하고, 상위 75%는 비교 과정에서 나온 자연어 비평을 반영하여 답안을 수정한다. 하위 25%는 탈락시킨다.

3단계: 최종 선택

살아남은 답안들에 대해 10번의 쌍별 비교를 추가로 실시하고, 최종 Bradley-Terry 랭킹 1위를 정답으로 선택한다.

핵심 통찰은 쌍별 비교가 개별 점수제보다 훨씬 정확하다는 것이다. 500쌍 진단 테스트에서 쌍별 비교 정확도가 86%인 반면, 개별 점수제는 59%에 그쳤다. "A와 B 중 어느 것이 나은가?"가 "A는 몇 점인가?"보다 LLM에게 훨씬 쉬운 과제인 셈이다.

주요 결과

경쟁 프로그래밍

  • Gemini 3.1 Pro의 Codeforces Elo: +405점 향상 (8라운드, 약 27분 소요)
  • 최난도(Hard) 문제 정답률: 11% → 50%
  • 문제당 평균 API 호출 수: 약 285회

교차 모델 전이

재조정 없이 다른 Gemini 모델에도 적용했을 때:

  • Gemini 3 Flash: pass@1 +30점
  • Gemini 2.5 Pro: pass@1 +22점

기존 방법과의 비교 (동일 연산 예산, 300회 호출)

  • 개별 점수제 최난도 정답률: 41%
  • OpenDeepThink 최난도 정답률: 50% — 순수 선택 품질 차이로 9%p 격차

한계가 드러난 영역

HLE 벤치마크의 객관적 도메인(수학, 물리, 생물)에서는 +517점 향상이 있었지만, 주관적 도메인(인문학, 사회과학)에서는 -2530점 하락했다. 쌍별 비교의 정확도가 심판 LLM의 해당 분야 역량에 직접 의존하기 때문이다.

의미와 전망

OpenDeepThink의 가장 중요한 시사점은 추론 성능 향상의 새로운 축을 제시했다는 것이다. 기존의 '더 긴 사고', '더 큰 모델'에 더해, '더 넓은 탐색 + 더 정확한 선택'이라는 경로가 실용적으로 유효함을 보여줬다.

연구진은 흥미로운 부가 발견도 보고했다. 쌍별 비교에서 "어디가 틀렸는지"를 지적하는 비평이 답안 수정의 핵심 동력이며, 긍정적 피드백은 거의 기여하지 않았다. 가장 큰 성능 향상은 0→1세대에서 일어나고, 2세대 이후에는 포화된다. 같은 연산 예산이면 더 많은 초기 답안을 생성하는 것이 더 깊은 비교를 수행하는 것보다 유리하다.

경쟁 프로그래밍, 수학 증명, 코드 디버깅 등 객관적 검증이 가능한 과제에서 즉시 활용 가능하다. 다만 문제당 약 285회의 API 호출이 필요하므로 실시간 응답이 아닌 배치 처리나 고가치 문제 해결에 적합하다.

Gemini 계열 모델에서만 검증되었으며, 다른 아키텍처(GPT, Claude, Llama 등)로의 전이성은 아직 미확인이다. 주관적 판단이 필요한 영역에서 성능이 하락하는 점은 심판 모델의 도메인 역량이 병목이 됨을 시사한다.
